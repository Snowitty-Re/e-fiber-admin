# ADR-0004: 采用单租户一仓一部署（预留多租户扩展位）

- 状态：Accepted
- 日期：2026-06-18
- 关联：`docs/00` §7、`docs/02` §1、`docs/03` §1.2

## 背景

产品定位为「快速建立外贸独立站」，需在「单租户简单可控」与「多租户 SaaS」间取舍。SaaS 复杂度高（数据隔离、租户上下文、计费），与「快速建立」MVP 目标不符。

## 决策

采用**单租户一仓一部署**：一个 admin + 一个 worker 进程对应一个商户站点，中间件 Docker 化。「快速建立」靠最小配置 + 一键 `make up && make api` 实现，而非 SaaS 隔离。

数据层**预留** `tenant_id` 字段（所有表，MVP 不实现隔离逻辑），为未来平滑升级多租户留扩展位。

## 备选

| 方案 | 否决理由 |
|---|---|
| 多租户 SaaS（tenant_id 行级隔离 / schema 隔离） | 架构复杂度高，偏离 MVP「快速建立」，隔离/计费/上下文成本大 |
| 单租户无扩展位 | 未来升级多租户需大改表结构 |

## 后果

- 正面：架构简单、易维护、易部署、MVP 快速交付。
- 负面：多商户需多套部署（运维成本随商户数增长）。
- 升级位：`tenant_id` 预留 + 接口层可加租户上下文中间件，未来可升级行级隔离的多租户。
